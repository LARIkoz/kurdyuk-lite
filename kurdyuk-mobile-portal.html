8 фаз с жёсткими gates

Ни одну фазу нельзя пропустить. Ни одну нельзя начать без confirmation предыдущей. Каждая фаза имеет входные условия и output артефакты.

Pipeline Flow — связи и петли обратной связи
P0 — Discovery + Market Research

Четыре опоры до старта: что строим / для кого / что НЕ строим / первый успех. Без всех четырёх Opus не выходит из Discovery. ASO keyword research влияет на naming уже здесь. Конкурентные скриншоты анализируются сразу — Gemini bg парсит топ-10 конкурентов.

brief.md research.md reference-screenshots/ compliance-checklist.md libraries.md gotchas.md
P1 — Xcode Setup + Foundation

Самая предсказуемая фаза: 90% автоматизировано. XcodeGen → project.yml, PrivacyInfo.xcprivacy, Info.plist, entitlements, Package.swift, Analytics init, Fastfile. Nested CLAUDE.md структура создаётся здесь. Human: 5 минут на team + auto signing + первый build.

project.yml CLAUDE.md Sources/CLAUDE.md Tests/CLAUDE.md Fastfile
P2 — IA + Data Model (Simple: skip)

Список экранов, navigation flow, data model sketch, paywall placement. Simple apps (3 экрана): пропускать. Standard: 30 мин с Opus. Complex (10+ экранов): HTML wireframe + Codex review обязательны. Неправильная IA = переписывать navigation на P4.

ia.md data-model.md wireframe.html
P3 — Design Exploration (РАЗВИЛКА)

Три режима: Mode A Quick (30 мин, 7-8/10), Mode B Boosted (2-3 часа, 9/10), Mode C AI Polish Loop (Gemini как critic, max 5 итераций). Category-aware referencing: Productivity vs Wellness vs Data — разные визуальные референсы.

DesignSystem.swift design-brief.md Cookbook.md
P4 — Implementation

Sonnet кодирует по плану. Экраны 1-2: hardcode. Экран 3: EXTRACT DesignSystem.swift. Niche KB queries (gotchas из dev_kb). Widgets, Paywall, Analytics, System APIs. Все states: empty, error, offline. Per-screen design-checklist.md. Codex bg review каждого major блока.

*.swift design-checklist.md
P5 — Polish (30–50% времени!)

Анимации, haptics, dark mode, empty states, app icon, launch screen, sound design. Accessibility: Dynamic Type, VoiceOver, contrast ratios. Это где создаётся "магия" — разница между "работает" и "хочется показать друзьям". Gemini bg делает Visual QA: сравнивает скриншоты с design brief.

Assets.xcassets accessibility-checklist.md
P6 — ASO + Submit

App Store metadata через fastlane deliver. Screenshot generation: simctl + device frames. Privacy nutrition labels. Compliance re-check (checklist из P0). TestFlight → App Store. Opus ревьюит метаданные на keyword density и conversion rate.

metadata/ screenshots/ Deliverfile
P7 — Iteration & Analytics

Analytics → backlog. Reviews → tasks v1.1. A/B тесты. Revenue metrics → pricing/features. Ключевое: P7 не завершает цикл — он его замыкает. P7 → P0: новый brief для v1.1. Это единственный способ не остановиться на "запустил и забыл".

v1.1-brief.md analytics-report.md
"Jordi Bruin: '2-2-2' — 2 часа PoC, 2 дня рабочая версия, 2 недели Store. Мой pipeline длиннее, потому что между Design и Implement живут Agent Skills, а после Polish — Mode C loop с Gemini как независимым критиком. Без этих слоёв получается 7/10 продукт, а не 9/10."
AI Automation

AI Automation Map

Что делает AI, что делает человек. Ни один агент не работает на всех фазах — каждая модель используется там, где у неё настоящее преимущество.

Модель → Фаза → Задача
Почему Opus для Discovery и планирования

Discovery — самая дорогая ошибка. Если задать неправильный brief, все последующие фазы работают не в ту сторону. Opus стоит дороже, но думает глубже: он задаёт правильные вопросы, ловит противоречия в требованиях, предлагает альтернативы которые Sonnet бы не заметил. Экономить на думающей фазе — значит платить вдвойне на Implement.

Почему Sonnet для кодирования

Sonnet быстрее и дешевле на повторяющихся задачах, где контекст уже задан CLAUDE.md и Skills. "Напиши SwiftUI экран по этому дизайну" — это не думающая задача, это точное выполнение. Sonnet делает это лучше и быстрее. Opus здесь избыточен и в 5-10 раз дороже.

Почему Gemini background для research

Gemini CLI запускается как background sub-agent через Bash (не Agent tool — там permissions bug #25520). Свежие данные из интернета, PDF документы, 2M context window. Для competitor research и compliance scan это незаменимо: Claude Code знает данные до августа 2025, а Apple Review Guidelines меняются каждый квартал.

Почему Codex background для code review

Codex (OpenAI gpt-5.4-mini) запускается параллельно с основной сессией — Claude пишет код, Codex ревьюит предыдущий блок. Это не rubber-stamp review: Codex ловит actor isolation ошибки, неправильные SwiftData queries, потенциальные App Store violations. gpt-5.4-mini достаточно хорош для review и стоит дешевле.

Почему прогрессивная загрузка, а не "всё сразу"

Claude Code видит весь контекст окна как равнозначный. Если в Discovery загрузить Accessibility skill, модель начнёт думать об accessibility constraints когда нужно думать о продуктовом brief. Прогрессивная загрузка — это контроль внимания модели, не просто экономия токенов.

Nested CLAUDE.md — бесплатная модулярность

Claude Code автоматически подхватывает nested CLAUDE.md — бесплатная модулярность без кода. Root CLAUDE.md: build system, .pbxproj=never. Sources/CLAUDE.md: SwiftUI + Concurrency правила. Tests/CLAUDE.md: Testing Pro patterns. Модель видит только релевантные правила для текущих файлов.

Структура nested CLAUDE.md
ProjectRoot/ ├── CLAUDE.md ← build cmds + ".pbxproj никогда" + --quiet flag ├── Sources/ │ ├── CLAUDE.md ← SwiftUI Pro (P3+) + Concurrency Pro (P4+) + Accessibility (P5) │ └── Models/ │ └── CLAUDE.md ← SwiftData Pro (P4+) └── Tests/ └── CLAUDE.md ← Swift Testing Pro (P4+)

Каждый раз когда Sonnet открывает файл в Sources/Models/, он автоматически видит SwiftData Pro rules. Это не магия — это правильная инженерия контекста.

Design Strategy: 3 режима

Единственная фаза с ветвлением. Три режима — три разных бюджета времени и денег. Выбор определяет потолок визуального качества продукта.

Category-aware referencing

Разные категории требуют разных визуальных референсов. Productivity: Things 3, Fantastical — чистота, информационная плотность. Wellness: Calm, Headspace — мягкость, whitespace, природные текстуры. Data/Technical: Car Scanner, OBD Fusion — dense UI, monospaced fonts, status indicators. Claude знает категорию из brief.md и автоматически использует правильные референсы.

Cookbook.md — библиотека рецептов

Каждый завершённый дизайн → рецепт в Cookbook.md. Формат: нишевый контекст + что сработало + почему + snippet. После 5-6 приложений Cookbook становится надёжным source of truth. Sonnet читает Cookbook в начале P3 и не изобретает колесо. Это compound interest от работы по pipeline.

Почему design generation только через Claude inline: Gemini CLI не генерирует изображения (только анализирует). Codex — text-only. Recraft API генерирует concept art, но SwiftUI translation требует Claude Sonnet inline. Нет shortcuts — design generation всегда inline.

Gemini bg парсит Apple Review Guidelines по категории (обновляются каждый квартал — Gemini видит актуальную версию). Параллельно app-store-scraper скачивает 1-star отзывы топ-конкурентов, фильтрует по ключевым словам: "rejected", "removed from App Store", "privacy issue", "compliance". Это реальные rejection signals, не абстрактные правила. Claude синтезирует оба источника в checklist.

Health & Fitness (RED — сложная категория)
Обязательные medical disclaimers ("not for diagnosis") · Запрещённые claims: "diagnose", "cure", "treat", "prevent" · HealthKit: обязательные privacy descriptions · Mental health apps: crisis resources & hotlines · Модерация 7–14 дней · Возможно human review
Finance (AMBER — специфичные требования)
Financial disclaimers ("not investment advice") · Регуляторные требования по стране · Subscription disclosures обязательны · No crypto trading без лицензии · Bank-level security если хранятся данные карт
Kids (PINK — строжайшие)
COPPA compliance обязательна · Parental gate для любых in-app purchases · Запрещены: behavioral advertising, analytics без consent · Возрастная граница 4+ требует минимальный UI complexity · Social features = автоматический reject для Kids category
Automotive / CarPlay
MFi requirements для hardware connectivity · CarPlay entitlement требует Apple Developer approval (2-4 недели) · Запрещены: сложные interactions во время вождения · OBD apps: не заявлять "emissions compliance" без сертификации
Productivity / Utilities (CYAN — относительно просто)
Стандартные privacy descriptions · Если clipboard access — объяснение в App Review Notes · Widget extensions: отдельные privacy declarations · Модерация 1–3 дня
Skill накопление
1-й health app → записались HealthKit gotchas (~30 мин research) · 2-й → добавились mental health disclaimers (~10 мин update) · 3-й → полноценная knowledge base (~5 мин check) · К пятому приложению в нише compliance research = 2 минуты
Adaptive Weight: Simple (таймер, Utility) → 2 checklist пункта, 2 минуты. Standard (Water Tracker, Health) → ~15 пунктов, 15 минут. Complex (OBD + CarPlay, Automotive) → полноценный checklist, 30 минут + Apple Developer portal work. Zero friction для простых apps. Полная защита для сложных.
Dev Knowledge Base

Niche Knowledge Base

Когда Claude пишет код для OBD-сканера, он не знает что ELM327 v1.5 теряет connection после 30 секунд idle. Niche KB решает эту проблему системно, а не через "добавь промпт с контекстом".

Три слоя знаний — как они соединяются
Почему gh search, а не RAG для live queries

gh search = 3 секунды, бесплатно, всегда актуально. RAG требует: pipeline сборки, ChromaDB, OpenAI embeddings (~$0.10/10K chunks), maintenance. Для "найти issue с этой библиотекой" gh search repo:OBDSwift state:closed label:bug отвечает за 3 секунды и возвращает актуальные данные с сегодняшней датой. RAG имеет смысл для большого структурированного corpus (VME, legal docs). Для dev gotchas — gh search выигрывает по всем параметрам.

Multi-Agent Routing

5 AI-агентов, каждый в своём слоте

Думающие фазы → Opus. Кодящая фаза → Sonnet. Web/PDF research → Gemini bg. Implementation review → Codex bg. Concept art → Recraft. Human → только approvals.

Фаза Opus inline Sonnet inline Gemini bg Codex bg CLI / Tools
Как запускать Gemini background

Gemini CLI запускается через Bash(run_in_background=true), а не через Agent tool. Причина: Agent tool не наследует permissions в VS Code (bug #25520). Bash наследует полные permissions основной сессии. OAuth auth через echo "Y" | GOOGLE_GENAI_USE_GCA=true. API key не используется — Gemini CLI работает только через OAuth.

Bash( command="echo 'Y' | GOOGLE_GENAI_USE_GCA=true gemini \\ -p 'Research: competitor analysis...' \\ -m gemini-2.5-pro -o text > /tmp/research.md 2>/dev/null", run_in_background=true )
Как запускать Codex background

Codex CLI (OpenAI) запускается аналогично через Bash background. Модель gpt-5.4-mini достаточна для review задач и расходует ~30% квоты vs gpt-5.4. gpt-5.4-nano возвращает 400 ошибку — не использовать. Флаги: -s read-only для review, -C для рабочей директории.

Bash( command="codex exec 'Review SwiftUI code: \\ check actor isolation, iOS 17+ API usage, \\ App Store compliance.' \\ -s read-only -C ./Sources/ \\ -o /tmp/codex-review.md", run_in_background=true )
Параллельный запуск: Gemini (research) + Codex (review) можно запустить одновременно как два background Bash в одном сообщении. Claude продолжает работу пока оба агента работают в фоне. Когда завершатся — читает оба результата и синтезирует. Это не последовательно, а параллельно — ключевое преимущество background pattern.
Tools & Infrastructure

Tools & Infrastructure

Инфраструктура pipeline. Каждый инструмент решает конкретную проблему. Shell Primed vs MCP — benchmark Sentry показывает что Shell = cheapest option.

Project Generation
XcodeGen
Генерирует .xcodeproj из декларативного project.yml. Решает проблему merge conflicts в .pbxproj (самый частый источник боли в iOS команде). Claude генерирует project.yml — человек никогда не трогает .pbxproj. Правило зафиксировано в root CLAUDE.md. Используется в P1.
MCP Tool · 82 tools
XcodeBuildMCP
82
82 инструмента для Xcode: build, test, simulate, screenshot, device management. "Critical for productivity" — plankenau.com после 8 часов до App Store. Known bug #846: периодические зависания на build. Workaround: xcodebuild через Shell как fallback.
MCP Tool · 20 tools
Apple Xcode MCP
20
Официальный Apple MCP сервер. 20 инструментов, менее богатый чем XcodeBuildMCP, но официальная поддержка = меньше breaking changes. Используется как fallback когда XcodeBuildMCP глючит.
Build Integration
Sweetpad
VS Code extension для Xcode build прямо из редактора. Позволяет Claude Code работать в VS Code и запускать Xcode build без переключения окон. Интеграция с xcodebuild через настраиваемые tasks. Используется когда MCP не доступен.
Screenshot · Mode C
simctl
xcrun simctl io booted screenshot /tmp/screen.png — скриншот симулятора за 1 секунду. Критичен для Mode C (AI Polish Loop): Sonnet строит → simctl screenshot → Gemini critiques → iterate. Без simctl Mode C невозможен.
ASO · P6
Fastlane
deliver для App Store metadata submission. snapshot для screenshot automation. Fastfile генерируется Claude в P1 — человек настраивает credentials один раз. Дальше P6 = fastlane deliver + human confirm. Снижает время P6 с 4 часов до 45 минут.
Shell Primed vs MCP — что выбрать

Исследование Sentry Engineering (реальный iOS проект, ~500 Claude Code commits) показало: Shell Primed подход — когда Claude использует xcodebuild напрямую через Shell без MCP — оказался самым дешёвым и надёжным вариантом. MCP инструменты добавляют overhead на каждый вызов и имеют дополнительные failure modes.

Review History

9 раундов ревью — ключевые решения

Pipeline эволюционировал через 9 rounds review. Каждый раунд менял что-то существенное. Это не итерации ради итераций — каждый pivot обоснован реальными проблемами.

Round 1–2
SwiftUI-first как baseline
Установили SwiftUI-only (не UIKit) как базовый принцип. Claude Code работает значительно лучше с SwiftUI — меньше legacy API confusion. Deprecated UIKit approaches убраны из skills.
Round 3
Monetization-first, а не feature-first
Paywall placement перенесено в P0 (Discovery) из P4 (Implement). Реализовывать экраны без понимания paywall placement = переписывать navigation дважды. Это изменило порядок IA в P2.
Round 4
3 режима дизайна вместо одного
Mode A/B/C появились как ответ на разные ниши и бюджеты. Один подход к дизайну не работает для всех: MVP нужен Quick, premium продукт — Boosted, competitive niche — Polish Loop.
Round 5
Compliance Research в P0, не P6
Compliance checklist переместился из P6 в P0. Обнаружить что нужен CarPlay entitlement (approval 2-4 недели) на этапе Submit = провал timeline. В P0 это решается заранее.
Round 6
Niche KB с gh search, не RAG
Рассматривался RAG pipeline (ChromaDB + OpenAI embeddings). Benchmark показал: gh search отвечает за 3 секунды, бесплатно, всегда актуально. RAG = overhead без преимущества для dev gotchas. VME RAG остался только для video transcripts.
Round 7
Progressive Quality Funnel

Opus · Phase 0
01. Discovery: четыре опоры перед стартом

Прежде чем что-то строить, фиксируем четыре вещи: что строим, для кого, что НЕ строим, и что считается первым успехом. Без всех четырёх ответов pipeline не начинается — Opus задаёт максимум три вопроса и ждёт. Это не бюрократия ради бюрократии: неправильный brief = все последующие фазы работают не в ту сторону.

ASO keyword research влияет на naming уже на этапе Discovery. Если целевое ключевое слово не влезает в название (максимум 30 символов) — это проблему нельзя решить потом без нового сабмита. Gemini bg парсит App Store за 20 минут: топ-50 apps по релевантным keywords, их названия, subtitle паттерны, review counts. Claude синтезирует: какие niches перегружены, какие недостаточно представлены.

Competitor rejection analysis — один из самых ценных шагов. app-store-scraper скачивает 1-star отзывы топ-10 конкурентов. Claude фильтрует по паттернам: "removed", "rejected", "stopped working", "privacy". Это реальные сигналы о compliance проблемах которые Apple уже ловила в этой нише. Гораздо лучше узнать об этом в P0, чем получить reject в P6.

Монетизационная модель решается в P0, не в P4. Это архитектурное решение, которое влияет на всё: onboarding paywall требует другого navigation flow чем hard gate после 3 дней использования. Subscription vs разовая покупка — разные юридические требования, разные StoreKit APIs, разный UX для upgrade flow. Решать это на этапе Implementation = переписывать половину экранов.

Четыре опоры brief.md:
1. Что строим (конкретно, не "fitness app")
2. Для кого (конкретный пользователь, не "люди которые хотят...")
3. Что НЕ строим (scope control — самое важное)
4. Первый успех (конкретная метрика: "100 скачиваний за первый месяц" или "первый paying user")
Adaptive Weight: Simple app (таймер, калькулятор) — Discovery занимает 30 минут. Standard app (Water Tracker с HealthKit) — 2 часа включая compliance research. Complex app (OBD Scanner с CarPlay) — полный день: compliance research включает CarPlay entitlement process (Apple approval 2-4 недели — это надо знать в P0, не в P6).

Sonnet · Phase 1
02. Xcode Setup: 90% автоматизировано

Phase 1 — самая предсказуемая фаза в pipeline. Практически всё делает Claude автоматически: XcodeGen генерирует project.yml без merge conflicts, Sonnet создаёт PrivacyInfo.xcprivacy с правильными API declarations, Info.plist с required keys, .entitlements с нужными capabilities, Package.swift со зависимостями из brief, Analytics init (Mixpanel или Amplitude), Fastfile для P6.

CLAUDE.md создаётся сразу и является инвестицией которая окупается на каждом шаге. Он содержит: команды сборки, флаг --quiet для xcodebuild (без него лог засоряет контекст), явное правило "никогда не редактировать .pbxproj напрямую", bundle ID, deployment target, список capabilities. Это инструкции для Sonnet которые он читает автоматически при каждом обращении к проекту.

Nested CLAUDE.md структура создаётся здесь, в P1, а не по мере необходимости. Sources/CLAUDE.md изначально пустой — Skills будут добавляться в P3 (SwiftUI Pro) и P4 (Concurrency, SwiftData). Важно создать структуру заранее: добавить skill в P3 = одна строка в уже существующем файле.

Human step в P1 — 5 минут: Xcode → Project → Signing & Capabilities → Team → Enable automatic signing → Build один раз для проверки. Это единственный шаг который физически нельзя автоматизировать — Apple требует human interaction для code signing setup. Claude может сгенерировать все файлы, но подписать сертификат должен человек.

Health Apps warning: Phase 1 для HealthKit-приложений = 1–2 дня, не "10 минут". Apple Developer portal требует: включить HealthKit capability, создать App ID с entitlements, обновить provisioning profile, пройти HealthKit review (дополнительный review process). Claude генерирует все файлы правильно, но Apple approval process = человеческое время. Планировать это в timeline с P0.

Opus · Phase 2
03. IA + Data Model: когда пропускать

Информационная архитектура — это список экранов, navigation flow, data model sketch и paywall placement. Не wireframe и не полноценный spec. Opus разрабатывает это в диалоге: "У нас onboarding paywall или trial первые 3 дня?" — это не дизайн вопрос, это архитектурное решение.

Simple apps (таймер, калькулятор, 3 экрана, один data type): пропускаем P2. Прямо в P3 Design. Standard apps (Water Tracker с HealthKit, 6-8 экранов): 30 минут с Opus — screen list, navigation graph, SwiftData schema, paywall placement. Complex apps (OBD Scanner, множественные data sources, 15+ экранов): HTML wireframe + Codex review обязательны. Неправильная IA в complex apps = переписывать navigation стек на P4 — это дорого (2-3 дня потерянного времени).

Paywall placement в P2 решает конкретную архитектурную задачу: нужен ли @EnvironmentObject с subscription state на всех экранах, или достаточно проверки в конкретных местах. Onboarding paywall = subscription state нужен до первого экрана приложения. Hard gate после trial = другая архитектура. Это решается в P2, а не обнаруживается в P4.

Opus Sonnet · Phase 3
04. Design режимы: детальный разбор

Design — единственная фаза с ветвлением. Выбор режима определяется в разговоре с Opus который читает brief.md и задаёт два вопроса: насколько конкурентна ниша визуально, нужен ли premium product feel. Ответы → Mode A, B, или C.

Mode A (Quick, 30 мин): app-store-scraper скачивает скриншоты топ-10 конкурентов. Claude анализирует: доминирующая цветовая схема, типографика (serif/sans, weight), whitespace density, characteristic animations (bounce vs spring vs linear), card styles. На выходе — три named style proposals: "Things 3 style" (minimal, iOS-native), "Calm style" (soft gradients, wellness), "DataUI style" (dense, technical). Human выбирает направление. Sonnet реализует через SwiftUI #Preview итерации — 20 минут до одобрения.

Mode B (Boosted, 2-3 часа): Pinterest API возвращает 50+ mood board images по нишевым keywords (например: "minimal productivity app UI", "clean iOS design 2026"). Recraft API генерирует custom concept art — абстрактные backgrounds, icons, textures которые соответствуют mood board. Gemini анализирует concept art: цвет, форма, характер. Sonnet матчит этот визуальный язык в SwiftUI компоненты. Результат: product feel который отличается от "стандартного iOS", а не очередной "SF Symbols + system colors" app.

Mode C (AI Polish Loop): Ключевое — Gemini выступает как независимый критик, не Claude который оценивает свою же работу. Gemini не знает историю задачи, не знает какие варианты уже рассматривались. Он смотрит скриншот и brief.md свежим взглядом. Цикл: Sonnet строит screen → xcrun simctl io booted screenshot /tmp/s.png → Gemini bg: "Compare screenshot vs brief requirements, list specific issues" → Claude читает critique → Sonnet применяет fixes → repeat max 5 раз. 5 итераций — hard limit: после 5 циклов diminishing returns, лучше сделать conscious design decision.

Category-aware referencing — конкретные примеры:
Productivity: Things 3, Fantastical, Notion Mobile — чистота, информационная плотность, iOS-native feel.
Wellness: Calm, Headspace, Streaks — мягкие градиенты, природные текстуры, large typography.
Finance: Revolut, YNAB — dashboard density, status colors (green/red), data visualization.
Automotive/Technical: Car Scanner, OBD Fusion — dark theme, monospaced data, status indicators.
Kids: Duolingo, Khan Academy Kids — rounded shapes, bright colors, playful animations.

Sonnet · Phase 4
05. Implementation: Progressive Skills Loading

Phase 4 начинается с полной загрузкой skills пакета: Sources/CLAUDE.md получает SwiftUI Pro (уже с P3) + Concurrency Pro + SwiftData Pro + Testing Pro. Каждый skill — не tutorial, а набор правил: "используй @Observable вместо ObservableObject", "actor isolation pattern для network requests", "SwiftData @Query в View, не в ViewModel".

Порядок реализации экранов важен: первые 1-2 экрана — hardcode значений для скорости. После третьего экрана становится видна паттерн design системы — тогда EXTRACT DesignSystem.swift: colors, typography, spacing, common components. Дальше все экраны используют DesignSystem. Без этого паттерна получается "лоскутное одеяло" где каждый экран немного отличается от другого.

Niche KB queries встроены в workflow: когда Sonnet пишет код который касается domain-specific функциональности (BLE, HealthKit, CoreData migration, CarPlay), автоматический query в ~/.claude/dev_kb/<niche>/gotchas.md. Если в KB нет ответа — gh search issues в релевантных repos. Это занимает 3-5 секунд, но предотвращает 2-3 часа debugging.

Все states обязательны с первого дня: empty state (нет данных), error state (network/permission fail), offline state (нет connectivity). Не "добавим потом в Polish" — добавлять потом = дороже. Empty states — это не просто "нет данных": это opportunity показать onboarding, объяснить value proposition. Car Scanner с empty OBD data state может показать "Подключите адаптер OBD-II" с animated illustration.

Codex bg review запускается параллельно с имплементацией каждого major блока (экраны, services, data layer). Claude пишет screens 3-5 → параллельно Codex ревьюит screens 1-2. К моменту когда Claude заканчивает screens 3-5, review screens 1-2 уже готов. Это не замедляет разработку — это "free" review без overhead.

Sonnet Gemini · Phase 5
06. Polish: 30–50% всего времени

Polish — недооценённая фаза. Большинство indie developers думают: "Polish — это немного анимаций, потом запускаем". Реальность: 30-50% общего времени разработки. Это где создаётся разница между "работает" и "хочется показать друзьям".

Анимации и переходы: matchedGeometryEffect для list→detail, withAnimation(.spring) с правильными параметрами (response: 0.5, dampingFraction: 0.75 для большинства UI transitions), @Namespace для hero animations. Sonnet хорошо реализует эти паттерны из SwiftUI Pro skill. Проблема: без visual feedback ("посмотри как это выглядит") может сгенерировать technically correct но visually dead animations. Mode C здесь критичен.

Haptics: UIImpactFeedbackGenerator для button taps, UINotificationFeedbackGenerator для success/error, UISelectionFeedbackGenerator для picker changes. Правило: haptics везде где есть meaningful interaction. Отсутствие haptics в 2026 = ощущение "дешёвого" приложения. Concurrency Pro skill содержит правильные patterns для async haptic feedback (main actor).

Dark Mode: не "добавить переключатель", а системный dark mode через color assets. Если в P3 использовались hard-coded hex colors (Mode A быстрый старт), P5 — время мигрировать в Color assets с dark/light variants. DesignSystem.swift обновляется.

Accessibility: Accessibility skill загружается в Sources/CLAUDE.md в начале P5. Dynamic Type — все текстовые элементы должны масштабироваться. VoiceOver labels для всех интерактивных элементов. Contrast ratios ≥ 4.5:1 для normal text, ≥ 3:1 для large text. Touch targets ≥ 44×44pt. Gemini Visual QA проверяет accessibility issues которые не видны в коде: overlapping elements, insufficient contrast на actual screenshots (не в теории).

App Icon и Launch Screen: App Icon — набор из 1024×1024 PNG, Sonnet может сгенерировать описание для Recraft API если нужен custom icon. Launch Screen через LaunchScreen.storyboard или Info.plist UILaunchScreen — без deprecated .storyboard patterns.

Opus · Phase 6
07. ASO + Submit: fastlane + metadata

App Store Optimization — недооценённая возможность. Название (30 символов), subtitle (30), keywords (100), description (4000). Opus оптимизирует по нескольким критериям: keyword density в searchable fields (title + subtitle + keywords = 160 total indexable chars), conversion rate в description (features vs benefits, social proof, clear value prop), локализация если нужна (Gemini помогает с переводом и cultural adaptation).

Screenshots — не afterthought. simctl делает базовые screenshots, но App Store screenshots требуют device frames и marketing copy. Fastlane frameit добавляет frames, Sonnet генерирует marketing copy для каждого screenshot ("Track your water intake" → "Never forget to drink water again"). Разница в conversion rate между plain screenshots и framed screenshots с copy — значительная.

Compliance re-check: checklist из P0 просматривается снова перед сабмитом. За время разработки могли добавиться новые permissions, capabilities, network requests. Каждый новый API usage требует privacy description. Sonnet проверяет все permissions в Info.plist против реального кода — нет orphan permissions, нет отсутствующих descriptions.

TestFlight → App Store: сначала TestFlight internal testing (сразу доступно), потом TestFlight external (Apple review ~24-48 часов), потом App Store submit. App Store review time: Utility 1-3 дня, Health/Finance 7-14 дней. Планировать в timeline с P0.

Opus · Phase 7
08. Phase 7: замыкание цикла

Phase 7 — не финальная фаза, а переход к следующему циклу. Без явного замыкания большинство indie apps остаются на v1.0 навсегда: запустили, получили несколько отзывов, забыли. Механизм замыкания: analytics данные через 2-4 недели после запуска → Opus синтезирует → новый brief.md для v1.1.

Analytics в P7 отвечают на конкретные вопросы: Какой onboarding step имеет наибольший drop-off? Какая фича используется больше всего? Где пользователи покидают paywall screen? Conversion rate freemium → paid? Среднее время до первого meaningful action? Эти данные не интуитивны — они удивляют в 80% случаев.

Reviews → tasks: Opus анализирует отзывы и формирует конкретные задачи. "App crashes when I..." → bug, high priority. "Would be great if..." → feature request, evaluate. "Doesn't support landscape" → platform feature, low effort. 1-star reviews всегда читать целиком — они содержат самую честную обратную связь.

v1.1 brief отличается от v1.0 в одном ключевом месте: теперь есть реальные данные. "Для кого" становится точнее: если 70% пользователей — women 25-35, это влияет на design decisions. "Первый успех" становится амбициознее: "200 платящих пользователей" вместо "первый paying user".

Code Quality
09. Claude пишет "средний" SwiftUI — как это фиксить

Indragie Karunaratne написал, что Claude Code выдаёт "significantly better than average developer" на SwiftUI (20K строк, реальный проект). Это правда. Но "better than average" ≠ production. Есть системные проблемы которые повторяются из проекта в проект без правильных skills.

iOS 16 defaults: без явного контекста Claude использует ObservableObject, @StateObject, @Published, foregroundColor (deprecated) вместо iOS 17+ API. SwiftUI Pro skill исправляет это на уровне контекста: модель видит правила до того как начинает писать. "Используй @Observable macro вместо ObservableObject" — одна строка в skill которая экономит 30 минут debugging в каждом проекте.

Swift Concurrency: путается в actor isolation, неправильно использует @MainActor, создаёт potential data races с shared mutable state, использует deprecated DispatchQueue.main.async вместо await MainActor.run. Concurrency Pro skill содержит паттерны из Swift Evolution proposals, не из устаревших tutorial sites.

.pbxproj corruption: Claude иногда пытается редактировать project file напрямую. CLAUDE.md с явным правилом "никогда не редактировать .pbxproj — использовать XcodeGen для изменений" полностью закрывает эту проблему. XcodeGen regenerates project из project.yml.

Unit tests: без Testing Pro skill генерирует тесты которые всегда проходят (тестируют реализацию, а не поведение). С Testing Pro пишет тесты с реальными assertions, правильным arrange-act-assert, async testing patterns.

Vinylogue case study (twocentstudios.com, ObjC→Swift rewrite): 7 дней, +11,275 −8,249 строк, 52 файла. $353 API cost (реально $20 на подписке). Что работало: xcodebuild self-fix loop, scaffolding. Что не работало: visual design iteration без screenshot feedback, unit tests "flawed and useless", iOS 16 defaults. Вывод: nested CLAUDE.md файлы решающе улучшают качество — это не опционально.

Architecture
10. Nested CLAUDE.md: бесплатная модулярность

Claude Code автоматически подхватывает nested CLAUDE.md файлы — это бесплатная модулярность без дополнительного кода. Каждый CLAUDE.md загружается когда Claude работает с файлами в соответствующей директории. Это как scoped linting rules, но для AI context.

Структура знаний должна соответствовать структуре кода. Root CLAUDE.md: build system, project conventions, .pbxproj rule, bundle settings. Sources/CLAUDE.md: SwiftUI conventions, Concurrency patterns, iOS version targets. Sources/Models/CLAUDE.md: SwiftData schemas, migration patterns, query optimization. Tests/CLAUDE.md: testing patterns, async testing, mock strategies.

Каждый раз когда Sonnet открывает файл в Sources/Models/, он автоматически видит SwiftData Pro rules. Не нужно каждый раз объяснять контекст — правила уже в контексте. Это не магия, это правильная инженерия контекста. Compound effect: с каждой фазой rules становятся более специфичными для задачи.

ProjectRoot/ ├── CLAUDE.md ← build cmds + ".pbxproj никогда" + --quiet + bundle ID ├── Sources/ │ ├── CLAUDE.md ← SwiftUI Pro (P3+) + Concurrency Pro (P4+) + Accessibility (P5) │ └── Models/ │ └── CLAUDE.md ← SwiftData Pro (P4+) + migration patterns └── Tests/ └── CLAUDE.md ← Swift Testing Pro (P4+) + async patterns + mocking

Dev Knowledge
11. Dev Knowledge: gh search vs RAG — правильный выбор

Рассматривался RAG pipeline для dev knowledge (ChromaDB + OpenAI text-embedding-3-small, тот же stack что в VME). Benchmark показал: для dev gotchas — это overkill.

gh search отвечает за 3 секунды, всегда актуально, бесплатно. gh search issues "ELM327 disconnect" repo:nickelc/SwiftOBD2 state:closed возвращает актуальные issues с датой сегодня. RAG потребовал бы: build pipeline, ChromaDB setup, OpenAI API calls (~$0.10/10K chunks), scheduled re-indexing когда repos обновляются, maintenance при schema changes.

RAG имеет смысл для VME (Video Methodology Extractor): corpus из 460 MP3 транскриптов (300K+ токенов), семантический поиск нужен, данные не меняются после ingestion. Для dev knowledge: corpus маленький, данные в GitHub уже indexed и searchable, свежесть критична.

Niche KB файлы (gotchas.md, libraries.md) накапливают curated knowledge между проектами в нише. Это не замена gh search — это его complement: gotchas.md содержит уже проверенные workarounds, gh search находит новые issues которые появились с последнего app в нише.

Target Apps
12. Целевые приложения для pipeline

Pipeline оптимизирован для indie apps с конкретными характеристиками: нишевое решение, понятная монетизация, ограниченный scope. Не enterprise app, не social network, не game.

Хорошо подходит
OBD Scanner / Car Diagnostics · Water / Habit Tracker · Timer / Focus (Pomodoro) · Калькулятор нишевый (mortgage, tip, unit converter) · Simple journal · Sleep tracker · Glucose / health metric tracker · Expense tracker · Recipe converter
Плохо подходит / требует адаптации
Social networks (real-time, moderation) · Complex games (UIKit, SpriteKit, Metal) · Enterprise apps (complex auth, compliance) · Apps с backend (отдельный pipeline нужен) · Marketplace / two-sided platforms · Apps с real-time collaboration

Sebastian Röhl (indie iOS dev) как benchmark: Simple utility apps, нишевые решения, subscription или premium. Один человек, от идеи до $1K MRR за 1-3 месяца. Это реалистичная цель для pipeline: не "стать следующим Notion", а "решить конкретную проблему для 500 paying users которые платят $2.99/месяц".
