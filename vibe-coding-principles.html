Вход UPD
01. Ошибка #0: начать без контекста

Открыл Claude. Написал "сделай мне трекер воды на Swift". Получил 200 строк кода, который компилируется, но не нужен. Знакомо?

Это ошибка #0 — самая распространённая и самая дорогая. Не баг в коде. Не плохой промпт. А отсутствие контекста, в котором AI должен работать.

Чатбот без контекста — это случайный джуниор, который не знает: какой у тебя проект, какие API ты используешь, какие решения ты уже принял, что ты пробовал и что не сработало. Он будет генерить generic output. Правильный синтаксис, нулевая ценность.

Голый промпт → generic output. Без CLAUDE.md AI не знает твои правила. Без skills — выдумывает endpoints. Без memory — забывает кто ты. Без pipeline — перескакивает к коду до того, как ты определил что строишь. Контекст — это не "nice to have". Это разница между "AI-помощник" и "AI-агент, который реально делает работу".

Вся система, описанная в этой статье, существует ради одного: превратить generic AI в специализированного агента. CLAUDE.md задаёт правила. Skills дают знания API. Memory хранит решения. Pipeline не даёт перескакивать. Run folder держит состояние. Каждый слой добавляет контекст. Контекст делает AI полезным.

Формула:
Голый Claude = умный чатбот, средний результат
Claude + CLAUDE.md = агент с правилами
Claude + CLAUDE.md + Skills = агент, который знает твои API
Claude + CLAUDE.md + Skills + Memory = агент, который помнит кто ты
Claude + CLAUDE.md + Skills + Memory + Pipeline = оркестратор

Каждый слой — это контекст. Чем больше контекста — тем точнее результат. Не "умнее модель" — а "больше контекста". Context > Model.

Практический вывод: если ты только начинаешь — не настраивай всё сразу. Начни с CLAUDE.md (правила проекта). Потом добавь 1-2 skills (API, с которыми работаешь). Потом memory. Потом pipeline. Каждый слой — ощутимый скачок качества. Но даже одного CLAUDE.md достаточно, чтобы AI перестал быть "случайным джуниором" и стал "джуниором, который знает проект".

Философия
02. Оркестрация: почему я не пишу код

Boris Tane сказал: "Never let Claude write code until you've reviewed and approved a written plan." Я согласен, но для меня это только начало.

Моя роль — не писать код и не просить одну модель написать код. Моя роль — дирижировать пятью AI-моделями, каждая из которых делает то, в чём она лучше остальных. Claude Opus думает и принимает архитектурные решения. Sonnet пишет код быстро и дёшево. Gemini CLI лезет в интернет, читает PDF, анализирует скриншоты. Codex CLI ревьюит план и код с позиции "а это вообще реализуемо?".

Это не мультитулинг ради мультитулинга. Это специализация. Когда я просил Opus писать код — он был медленный и дорогой. Когда просил Sonnet думать об архитектуре — он соглашался со всем и не видел проблем. Когда просил Claude искать в интернете — он галлюцинировал ссылки. Каждая модель хороша в своём слоте. Плоха — во всех остальных.

Ключевой insight: вайбкодинг — это не "AI пишет код за тебя". Это операционная система, где ты — оркестратор. AI = лучший джуниор: делает 80% рутины, но без твоего контекста, вкуса и judgment — результат будет средним. Код — побочный продукт. Главный продукт — правильные решения.

Boris работает с одной моделью (Claude) в одной длинной сессии. Это работает для него — он Engineering Lead в Cloudflare, он знает архитектуру лучше любой модели. Ему достаточно одного Claude как исполнителя.

У меня другая ситуация. Я строю продукты с нуля — market research, gap analysis, iOS apps, data pipelines. Мне нужен не один исполнитель, а команда специалистов. И я собрал эту команду из AI-моделей.

Mindset
03. Ограничений нет — есть лень модели

Самое частое заблуждение: "AI не может это сделать". Может. Почти всегда может. Но он будет лениться — предлагать обходные пути, упрощать задачу, говорить "это невозможно" или "это выходит за рамки моих возможностей".

Это не ограничение. Это поведение по умолчанию. Opus на первый запрос даёт "безопасный" ответ. На второй — чуть лучше. На третий, когда ты давишь и объясняешь что конкретно нужно — он находит оптимальное решение, которое существовало с самого начала.

Паттерн "3 попытки":
Попытка 1: "Это сложно, попробуйте сделать вручную / используйте другой инструмент"
Попытка 2: "Ок, вот вариант, но он неоптимальный..." (работает на 70%)
Попытка 3: "Вот чистое решение" (то, что нужно было с самого начала)

Реальный пример: cookies. Нужно было скачать контент с сервиса, где авторизация через браузер. Первая реакция Claude: "Вам нужно залогиниться в браузере и скачать вручную." Второй запрос: "Можно попробовать через curl с cookie-файлом, но это ненадёжно." Третий: yt-dlp с --cookies-from-browser — одна команда, подхватывает cookies прямо из Arc, скачивает всё автоматически. 460 файлов. Без единого ручного клика.

Ещё пример: PDF. "Claude не умеет читать PDF." Не умеет — если просишь его напрямую. Но Gemini CLI читает PDF нативно. А yt-dlp качает с Яндекс Диска. А ffmpeg конвертирует что угодно. Ограничение одной модели — не ограничение системы. Нужно просто знать, какой инструмент дёрнуть.

Whisper на M1. "MPS не поддерживается для Whisper." Первая попытка — ошибка. Вторая — "используйте CPU". Третья — fp16=False и ssl._create_unverified_context — работает. 62x быстрее через Groq API, если локально не хочется возиться.

Принцип: когда AI говорит "невозможно" — это значит "я не нашёл с первого раза". Давишь — находит. Не давишь — остаёшься с ручной работой. Модель оптимизирует на "не ошибиться", а не на "решить задачу". Твоя работа как оркестратора — сдвинуть этот приоритет.

Это не только про Claude. Любая модель ленится:

Gemini обрезает длинные ответы без предупреждения. Workaround: разбить prompt на части или использовать flash для длинных ответов.

Codex отказывается работать с файлами вне рабочей директории. Workaround: флаг -C path/ для смены контекста.

Claude в subprocess теряет auth сессию. Workaround: Anthropic SDK + API key вместо claude -p через subprocess.

Антипаттерн: принять первый ответ как финальный. AI обучен быть helpful и safe. "Safe" = отказать, предложить ручной вариант, упростить. Ты должен быть настойчивым, конкретным и точным в запросе. Не "сделай как-нибудь" — а "сделай именно так, вот контекст, вот ограничения, не предлагай альтернатив — реши эту задачу".

Итог: ограничений нет. Есть лень модели, workarounds для известных багов и правильный инструмент для каждой задачи. 19 skills, 5 моделей, CLI-тулзы — если одна дорога закрыта, всегда есть другая. Оркестратор это знает. Пользователь одной модели — нет.

Рабочее место
04. VS Code + Claude Code: не расширение, а операционная система

Большинство людей думают о Claude как о чатботе. Окно, куда пишешь вопрос, получаешь ответ. Или как о расширении VS Code — автокомплит, inline-подсказки, Copilot-стиль.

Это не то, как я работаю.

Claude Code в VS Code — это не расширение. Это полноценный AI-агент, который имеет доступ ко всему: файловая система, терминал, git, интернет (через инструменты), запуск любых команд, чтение и запись файлов, создание процессов. Он не "помогает писать код" — он выполняет задачи. Разница как между калькулятором и компьютером.

Что может Claude Code как агент (а обычный чатбот — нет):
— Запустить Gemini CLI в background и продолжить работать
— Скачать 460 подкастов через yt-dlp с cookies из браузера
— Создать SQLite базу, наполнить данными, запустить миграции
— Прочитать 19 skill-файлов и автоматически выбрать нужный API
— Открыть URL в Arc, запустить npm build, прочитать логи ошибок
— Создать git branch, написать код, прогнать тесты, сделать commit
— Параллельно запустить Codex review и Gemini Visual QA в background

Вкладки и сессии. В VS Code у Claude Code есть вкладки — как вкладки в браузере. Каждая вкладка — отдельная сессия с отдельным контекстом. Я использую это так:

Вкладка 1: Teardown Suite — текущая задача Вкладка 2: Gap Pipeline — фоновый research Вкладка 3: Video Extractor — debug проблемы Вкладка 4: Quick task — одноразовая задача

Каждая вкладка помнит свой контекст в рамках сессии. Но контекст — конечный. Когда он заполняется, Claude сжимает старые сообщения (compression). Это нормально — если ты работаешь через файлы, а не через чат.

Permissions. VS Code спрашивает разрешение на каждое действие: чтение файла, запуск команды, запись. Можно настроить автоматическое разрешение для безопасных операций (чтение, grep, glob) и ручное подтверждение для опасных (запись, удаление, git push). Это не ограничение — это контроль.

Multiple working directories. Claude Code видит не только текущий проект — он может работать с несколькими директориями одновременно. У меня в конфигурации 8 рабочих директорий. Когда я говорю "проверь как это сделано в teardown-suite" — он идёт и читает код оттуда.

Persistence
05. Память, контекст и почему Claude помнит кто ты

Самый частый вопрос: "Как Claude помнит контекст между сессиями?" Ответ: файлы. Четыре слоя persistence, каждый для своего:

Слой 1: CLAUDE.md — operating contract. Живёт в корне проекта. Загружается автоматически в каждую сессию. Здесь: правила pipeline, model routing, annotation cycle, output style, запреты. Это как должностная инструкция для AI. Claude читает её первой и следует ей всю сессию.

// Пример из моего CLAUDE.md: Ты работаешь в режиме Kurdyuk Lite. Не начинай код, пока implementation-readiness не имеет verdict pass. plan.md — канонический implementation contract. Verification: не отмечай задачу ✅ без proof.

Слой 2: Memory — persistent файловая память. Не чат-история, а структурированные markdown-файлы. Четыре типа: user (кто я, как работаю), feedback (что делать / не делать), project (статусы, решения), reference (где что искать). Claude читает их при старте сессии и обновляет по ходу работы.

У меня сейчас 30+ файлов памяти. Claude знает: какие проекты активны, какие API gotchas найдены, что я предпочитаю русский язык, что не надо мокать базу в тестах, что stale plans нужно удалять. Это не "запомни" в чате — это файлы на диске, которые переживут любое количество сессий.

Важно: память — не замена контексту сессии. Память хранит cross-session знания (кто я, уроки, решения). Контекст текущей задачи живёт в run-файлах (state.md, plan.md, todo.md). Это разные вещи. Память = "кто ты и что знаешь". Run-файлы = "что делаешь прямо сейчас".

Слой 3: Skills — API-знания. 19 native skills — каждый в своём ~/.claude/skills/<name>/SKILL.md. Base URL, auth, endpoints, rate limits, pricing, gotchas. Загружаются автоматически по описанию — когда задача касается API, Claude находит нужный skill и знает как с ним работать.

Write-back: если в процессе работы Claude находит новый gotcha у API — он обновляет SKILL.md. В следующей сессии любой агент уже знает про эту проблему. Skills — живая документация, не статичная.

Слой 4: Run folder — состояние текущего проекта. .kurdyuk-lite/runs/<slug>/ — 9 файлов: state, brief, intent, invariants, research, plan, todo, readiness, review. Полный audit trail. Любой агент может войти, прочитать state.md и продолжить с нужного места.

Вход в новую сессию (3 шага):
1. MEMORY.md → кто я, что знаю, какие проекты активны
2. state.md → на какой фазе, что делаю, есть ли blocker
3. Продолжай с нужного места — как будто не уходил

Нет потери контекста. Нет "напомни что мы делали". Файлы помнят всё.

Hooks и автоматизация. Claude Code поддерживает hooks — shell-команды, которые запускаются автоматически на определённые события. Можно настроить: "после каждого коммита — запусти линтер", "перед каждым edit — проверь что файл не в .env". Hooks — это программирование поведения Claude, не просто промпты.

Settings. Три уровня: глобальные (для всех проектов), проектные (для конкретного репозитория), и пользовательские (мои личные правила). Permissions, model routing, env переменные, разрешённые команды — всё конфигурируется. Это не чатбот с настройками — это платформа.

Антипаттерн: полагаться на чат-историю. Чат = volatile. Compression съедает контекст. Новая сессия = чистый лист. Если важная информация живёт только в чате — она пропадёт. Всё ценное должно быть в файлах: CLAUDE.md, memory, skills, run folder. Чат — для текущего диалога. Файлы — для persistent знаний.

Infrastructure
06. Система skills: каждый API документируется один раз

Когда в проекте появляется внешний API — любой, от Groq до Pinterest — первое что я делаю: создаю skill. Не пишу код. Не экспериментирую вслепую. Создаю файл ~/.claude/skills/<api-name>/SKILL.md и заполняю его.

Зачем? Потому что AI галлюцинирует API endpoints. Уверенно. Он назовёт тебе /api/v2/search, которого не существует. Ты потратишь 30 минут на debug, прежде чем поймёшь что endpoint выдуман. Skill — это source of truth, который Claude читает вместо того, чтобы выдумывать.

~/.claude/skills/groq-api/SKILL.md --- name: groq-api description: Groq API — сверхбыстрая транскрипция через Whisper (217-228x realtime) user-invocable: false --- ## Base URL https://api.groq.com/openai/v1 ## Auth Authorization: Bearer $GROQ_KEY_1 Keys: /Documents/cursore/skill-prompts/apis/keys.env ## Endpoints POST /audio/transcriptions model: whisper-large-v3 file: binary (max 25MB) response_format: verbose_json ... ## Rate Limits 20 req/min, 2h audio/day per key Workaround: multi-key rotation ## Gotchas - Max file 25MB → split with ffmpeg - timestamps: word-level only with verbose_json - VERIFIED: 2026-03-19

Как создаётся skill: не из памяти модели. Из максимального research официальных источников. Процесс выглядит так:

Шаг 1: Gemini CLI (background) — идёт в интернет, находит официальную документацию API. Читает docs, quickstart, reference, changelog. Вытаскивает: base URL, auth, все endpoints с request/response, rate limits, pricing.

Шаг 2: GitHub repos — Gemini ищет SDK, примеры использования, issues. В issues — реальные баги, workarounds, edge cases, которых нет в документации. В примерах — рабочий код, не теоретические endpoints.

Шаг 3: Социальные сети и Reddit — опыт реальных людей. "У Groq rate limit 2h/day, но если ротировать ключи..." "Soniox context.terms принимает только английские произношения, транслитерации — через regex post-processing." Этого нет ни в docs, ни в GitHub — только в обсуждениях.

Шаг 4: Opus (inline) — синтезирует всё в структурированный SKILL.md. Убирает мусор, оставляет actionable знания: endpoints, gotchas, workarounds.

Результат: когда через 2 недели я (или другой проект) снова работаю с этим API — Claude автоматически подгружает skill по описанию и знает всё: endpoints, auth, gotchas. Ноль галлюцинаций. Ноль потерянного времени на re-research. У меня 19 таких skills и каждый экономит часы при каждом использовании.

Write-back: нашёл новый gotcha в процессе работы → тут же обновил SKILL.md, поставил новую дату VERIFIED. Skills — живая документация, не архив.

Экономика
07. Подписка vs API: почему CLI-модели выгоднее

Codex CLI (OpenAI) и Gemini CLI (Google) — это не API с оплатой за токены. Это CLI-инструменты, которые работают по подписке. ChatGPT Plus ($20/мес) даёт доступ к Codex CLI. Google AI Studio (бесплатно) даёт доступ к Gemini CLI.

Почему это важно: один run через Kurdyuk pipeline — это Research (Gemini) + Annotation (Codex) + Post-Review (Codex) + Visual QA (Gemini). Через API это стоило бы $5-15 за run. Через подписку — $0 сверх фиксированной платы.

Экономика sub-агентов:
— Gemini CLI: бесплатно (Google AI Studio OAuth). 2M контекст, web search, PDF, vision
— Codex CLI: $20/мес (ChatGPT Plus). GPT-5.4-mini, 30% квоты на code review
— Claude Code: $20/мес (Pro) или $100/мес (Max). Opus + Sonnet без лимитов
— Итого: $40-120/мес за полный AI-оркестр из 5 моделей
— Через API: те же задачи = $200-500/мес при активном использовании

Как это работает технически: Claude Code запускает Codex и Gemini как background sub-agents через Bash(run_in_background=true). CLI-инструмент наследует auth сессию пользователя (OAuth для Gemini, ChatGPT login для Codex). Никаких API ключей, никаких rate limits по токенам — только limits самой подписки.

Auth один раз: Gemini CLI — OAuth через браузер, один раз, потом работает. Codex CLI — логин ChatGPT, один раз. Дальше Claude запускает их сам, без твоего участия.

Архитектура
08. Сессии: почему файлы важнее чата

Boris предпочитает одну длинную сессию: "No performance degradation at 50% context window." Он прав — plan.md выживает auto-compaction, и в рамках одного проекта это работает.

Но я работаю над 8+ проектами параллельно. Teardown Suite, Gap Pipeline, Video Methodology Extractor, Expert RAG Portal, Kurdyuk Mobile, порталы, research. Длинная сессия на один проект — роскошь, которую я не могу себе позволить.

Поэтому мой pipeline — session-agnostic. Всё ценное живёт в файлах:

state.md — точка входа. Фаза, текущий шаг, blocker. Новая сессия = читай state.md.

Run folder — полный audit trail проекта. Brief, intent, invariants, research, plan, todo, readiness, review. Любой агент (или я через неделю) может войти, прочитать state.md и продолжить с нужного места.

Memory — persistent cross-session знания. Не дублирует run-файлы. Хранит: кто я, что предпочитаю, какие lessons learned, где какие API gotchas. Загружается автоматически в каждую сессию.

Skills (19 штук) — API-знания. Base URL, auth, endpoints, pricing, gotchas. Нашёл gotcha → обновил SKILL.md. В следующей сессии любой агент уже знает про эту gotcha.

Hard-won lesson: stale plan опаснее отсутствующего. Завершённый plan.md в контексте = AI думает работа не сделана и предлагает её снова. 3+ сессии подряд Claude предлагал batch regen, который давно завершён. Теперь правило: реализованный план → удалить файл. Plan = живой контракт, не архив.

Handoff protocol при смене сессии: обновить state.md → обновить memory (если есть cross-session value) → обновить skills (если нашёлся API gotcha) → cleanup (удалить stale). Вход в новую сессию: MEMORY.md → state.md → продолжай.

Итог: Boris оптимизирует для глубины одной сессии. Я оптимизирую для ширины — много проектов, много сессий, много агентов. Оба подхода работают. Выбор зависит от того, как ты работаешь.

Контроль
09. Дисциплина: не давай AI перескакивать

AI хочет писать код. Прямо сейчас. Ты ещё на фазе обсуждения, а он уже редактирует файлы. Ты задал вопрос про архитектуру, а он выдал "вот, я уже всё реализовал". Это самая опасная привычка AI-агентов — склонность перескакивать к следующему действию.

Типичный сценарий: ты утвердил план. AI написал код. Ты говоришь "хм, вот тут не так". AI: "Исправил! И заодно рефакторнул три соседних файла, добавил кэширование и переименовал API." Ты не просил. Но он уже это сделал. И откатить сложнее, чем было бы не делать.

Правило: фаза обсуждения ≠ разрешение на implement. "Что думаешь?" ≠ "делай". "Посмотри на это" ≠ "правь". AI должен показать что планирует — и ждать явного "делай" / "ок" / "приступай". Без этого слова — руки прочь от файлов.

Annotation cycle — главный защитный механизм. Между Plan и Implement стоят 1-6 циклов ревью. После каждого Claude спрашивает: "Нужен ещё цикл или план готов?" Не решает сам. Ты гоняешь циклы, пока план не станет таким, что доработки мелкие или их нет совсем. Только тогда — implement.

Типичная сессия annotation cycle: Цикл 1: Red Team → "scope drift в Phase 3, 2 лишних API" Claude обновляет plan → "ещё цикл?" Цикл 2: Codex (bg) → "endpoint /api/v2/sync не существует" Claude обновляет plan → "ещё цикл?" Цикл 3: Architect → "SQLite хватит, Postgres избыточен" Claude обновляет plan → "ещё цикл?" Цикл 4: User → "убери регистрацию, сделай проще" Claude обновляет plan → "ещё цикл?" Цикл 5: Мелкие правки, ничего structural → "план готов, переходи к readiness"

5 циклов — не бюрократия. Каждый цикл ловил реальную проблему. Без них я бы начал кодить с несуществующим endpoint, лишней базой данных и фичей, которую никто не просил. А потом откатывал бы 3 часа работы.

Когда остановиться: когда замечания мелкие (форматирование, naming) или их нет. Если каждый цикл находит structural issue — план ещё сырой. Если находит только cosmetic — план готов. Readiness Gate подтвердит.

Декомпозиция: дрилл вглубь

Ещё один аспект дисциплины — не принимать первый ответ за финальный. Opus выдал список из 8 улучшений для implement-фазы. Один пункт: "B6 — Fresh context per task". Звучит просто. Одна строчка в плане.

Я попросил расписать подробнее. Opus разложил B6 на 6 подзадач. Я увидел минусы в нескольких — попросил про каждый минус отдельно. Каждый минус развернулся ещё в 2-3 подзадачи. Итого: одна строчка плана → дерево из ~20 задач, часть которых нетривиальные.

B6 — Fresh context per task (выглядит как 1 задача) └── 6 подзадач после первого drill-down ├── Подзадача 2: минус → 3 варианта решения ├── Подзадача 4: минус → 2 архитектурных trade-off └── Подзадача 6: минус → нужен отдельный research Итого: ~20 реальных задач вместо 1 строчки

Техника: фиксируешь верхний уровень в план. Просишь углубиться в каждый пункт. Фиксируешь. Идёшь глубже. На каждом уровне Opus видит полное дерево — не теряет контекст и глубину.

Правило drill-down: если Opus выдал список > 3 пунктов — каждый пункт потенциально скрывает подуровень. Не принимай плоский список за финальную декомпозицию. Проси "распиши подробнее пункт N" хотя бы для тех, которые кажутся нетривиальными. Дешевле найти сложность сейчас, чем в середине implement.

Концепция ≠ инженерия

Ещё одна форма "перескакивания": AI смешивает ЧТО нужно с КАК реализовать. Ты формулируешь концепцию — а Opus тут же начинает вставлять промпты, алгоритмы, пороги, SQL-запросы. Документ из "конституции" превращается в implementation spec. И тогда при изменении реализации ломается смысл.

Реальный пример: Gap Pipeline — определение ниши. Первые версии смешивали "что такое ниша" (user intent, tier-система, false positives) с "как обнаруживать" (какие keywords фильтровать, какие пороги ставить, какие промпты давать LLM). Результат: каждый раз когда менялся алгоритм — ломалось определение. Промпт переписан → пороги поехали → тиеры перестали совпадать → всё переделывать.

Решение: два документа. Концепция (конституция) — только ЧТО и ПОЧЕМУ. Pipeline spec — только КАК. Концепция не меняется при смене алгоритма. Алгоритм не ломает смысл.

Концепция (конституция): "Ниша = user intent. Tier: PROVEN ≥$10K, EMERGING $1K-10K, OPPORTUNITY <$1K, GAP = 0 apps. Принцип: не фильтровать — тиеровать. Ничего не выбрасывается." Pipeline spec (инженерия): "Phase 3: LLM classify → prompt v4.2 → threshold 0.7 → SQLite JOIN → Louvain clustering resolution=1.2" Поменял Louvain на DBSCAN? Концепция не тронута. Поменял threshold? Тиеры те же. Переписал промпт? "Не фильтровать — тиеровать" по-прежнему закон.
Правило: если в документе есть и "зачем это нужно" и "SELECT * FROM niches WHERE..." — документ смешанный. Разделяй. Концепция = конституция, инженерия не имеет права её ломать. Никогда.

Advanced
10. Swarm: много агентов на одну задачу

Стандартный паттерн: один агент делает одну задачу. Но что если тебе нужно 10 разных мнений на один и тот же вопрос?

Пример: ты сделал UI для нового экрана приложения. Вместо того чтобы спросить одного Claude "как тебе?" — запусти 5-10 агентов параллельно, каждый с разной persona:

# Запускаем 6 агентов параллельно: Agent("Casual user, 25, uses phone 4h/day. Look at this screen. Is it obvious what to do? What confuses you? Be honest.") Agent("Power user, developer, very picky. Review this UI. What's broken? What's slow? What would you change?") Agent("Mom, 55, not tech-savvy. Can you figure out how to track water here? What's unclear?") Agent("Designer, worked at Apple. Rate this UI 1-10. What breaks iOS HIG? What's the weakest element?") Agent("Competitor analyst. Compare this to top 3 water trackers. What's missing? What's better?") Agent("Accessibility tester. Can a VoiceOver user navigate this? Dynamic Type breaks anything?")

Это не абстракция. Каждый агент — реальный sub-agent с контекстом. Он видит скриншот, читает brief, понимает задачу. И отвечает из своей перспективы. 6 агентов = 6 разных взглядов на один экран. За 2 минуты. Бесплатно (подписка).

Где это работает:
— UI feedback: 5-10 persona-агентов дают "user testing" без реальных пользователей
— Copy review: агенты-"читатели" с разным уровнем грамотности проверяют текст
— Architecture review: агент-"junior", агент-"senior", агент-"SRE" смотрят на один план
— Market research: агенты ищут информацию в разных нишах параллельно
— ASO: агенты-"пользователи" оценивают скриншоты и описание в App Store

Это не A/B тест. Это A/B/C/D/E/F тест, где каждый "пользователь" — AI-агент с настроенной persona. Дешевле, быстрее и доступнее, чем UserTesting.com. Не заменяет реальных пользователей на 100% — но закрывает 80% очевидных проблем до того, как реальный пользователь их увидит.

Параллельность: все агенты запускаются одновременно. Не последовательно. Claude Code поддерживает множественный запуск Agent tool в одном сообщении. 5 агентов = 5 параллельных процессов = результат за время самого медленного, не за сумму всех.

Принцип swarm: когда тебе нужно не "правильное решение", а "спектр мнений" — не проси одного агента думать за всех. Запусти много агентов с разными ролями. Один пропустит проблему. Шестеро — маловероятно. Тот же принцип что в Annotation Cycle, но масштабированный.

Не угадывать. Не гуглить 2 часа. Запустить арбитраж.

Ревью на ревью: тот же принцип работает для ревью. Система выдала code review — "всё ок, мелкие замечания". Не верь. Запусти второго агента: "Вот code review. Ты мета-ревьюер. Что пропущено? Какие замечания поверхностные? Где ревьюер был ленив?" Мета-ревью ловит то, что первый ревью пропустил. Ревью на ревью — не паранойя, а calibration.

Когда использовать арбитраж:

— Архитектурное решение с 2+ равносильными вариантами
— Выбор технологии (какая БД, какой фреймворк, какой API)
— Дизайн-решение, когда оба варианта "нормальные"
— Любой момент, когда ты ловишь себя на "а хрен его знает"

Brainstorm: два режима

Арбитраж решает "A или B?". Но brainstorm — это шире. У меня два режима, оба оформлены как native skill (Claude выбирает автоматически).

Режим A: Evaluation (default) — идея есть, нужно проверить.
Триггеры: "норм ли это", "разберём", "ревью идеи"

Flow: зафиксировать идею → reframing (challenge 1 premise) → trade-off таблица → Red Team agent (be brutal) → cross-model review (Claude + Codex параллельно) → verdict: do / rework / kill. Max 3 итерации — если после 3 нет clean verdict, идея слишком сырая.
Режим B: Socratic Rebuild — упёрся, вариантов нет.
Триггеры: "упёрся", "не складывается", "пересобрать концепцию"

Flow: Claude = Сократ, не лектор. Не даёт ответ — задаёт вопросы.
B0. Загрузка контекста (прочитать ВСЁ: run-файлы, memory, код, историю решений)
B1. Диагностика тупика: "Что конкретно не складывается?" (1-2 вопроса)
B2. Декомпозиция: разбираем на части вопросами (3-5 вопросов, по одному за сообщение)
B3. Переформулировка: "Если описать одним предложением — что это?" (2-3 вопроса)
B4. Проверка на прочность: "Что может пойти не так?" (2-3 вопроса)
B5. Фиксация: итог в 3-5 пунктов → записать в brief/intent/plan

Почему Socratic Rebuild, а не "предложи решение": когда ты говоришь "предложи" — Opus берёт твои текущие assumptions и ищет решение внутри них. Проблема в том что assumptions могут быть неправильными. Наводящие вопросы вскрывают assumptions: "Почему ты решил что нужен real-time? Действительно ли пользователь видит разницу между 1 секундой и 50мс?" — и вдруг оказывается что polling достаточно.

Ключевое правило: по одному вопросу за сообщение. Минимум 8-12 вопросов за сессию. Не торопиться к ответу. Если пользователь говорит "не знаю" — это ценный сигнал, переформулировать вопрос проще. Старая концепция = антипаттерн, не шаблон для починки.

Сигналы что нужен режим B:
— Костыли множатся быстрее чем фичи
— Каждый fix ломает что-то рядом
— Ты 3й раз переписываешь один и тот же модуль
— Opus "буксует" — даёт одинаковые ответы разными словами

Это значит проблема не в коде, а в концепции. Не чини код — пересобери концепцию через вопросы.

Quality
13. Итеративная охота за багами

AI написал код. Тесты проходят. Ты думаешь — готово. Нет. Это только начало.

Один проход code review ловит очевидные баги. Два прохода — находят logic errors. Три — находят edge cases. Четыре — находят integration issues. Каждый следующий проход ловит то, что предыдущие пропустили. Потому что каждый проход смотрит свежими глазами.

Реальный пример: Video Methodology Extractor. Проект прошёл через десятки итераций ревью. Вот что находилось на каждом уровне:

Уровень 1: Очевидные баги
— Whisper на MPS падал с ошибкой. Fix: fp16=False. Нашли на первом прогоне.
— SSL-сертификат не проходил на macOS. Fix: ssl._create_unverified_context.

Уровень 2: Плохой research → баги в логике
— Не знали что в подкасте 2 спикера, не один. Весь transcription pipeline был заточен под single speaker. Пришлось добавлять diarization (разделение спикеров). Это не баг в коде — это баг в research: не нашли что в аудио несколько голосов.
— Не знали что Soniox поддерживает context window для словаря (до 8000 tokens). Первая интеграция не использовала context.terms. Результат: "AdMob" транскрибировался как "ad mob", "Applovin" как "app lovin". Пришлось добавлять custom vocabulary.

Уровень 3: Плохой skill → неправильные вызовы API
— Skill для Soniox не документировал что context.terms принимает только английские произношения, не транслитерации. Писали "АдМоб" в terms — Soniox игнорировал. Нужно "AdMob". Это нюанс, которого нет в документации — только в issues на GitHub.
— Skill для Groq не документировал max file size 25MB. Файлы по 40MB падали молча. Fix: ffmpeg split перед отправкой. Skill обновлён.

Уровень 4: Regex и post-processing edge cases
— "адмоб" → "AdMob" работало. Но "admob" → "AdMob" — нет (другой регистр). Пришлось делать 3-tier regex: Tier 1 (always), Tier 2 (common), Tier 3 (context-gated).

Session Mgmt
16. Handoff, cleanup и гигиена сессий

Сессия Claude Code — не бесконечная. Контекст заполняется. Compression съедает детали. Через 40-50 сообщений модель начинает "забывать" начало разговора. Это нормально. Но к этому нужно быть готовым.

Handoff — протокол передачи состояния при завершении сессии. Не "закрыл вкладку и ушёл", а осознанная фиксация:

Handoff checklist (конец сессии): 1. STATE (всегда): - Обновить state.md: phase, current step, blocker - Run COMPLETE → status: complete, убрать "in progress" 2. MEMORY (если есть cross-session value): - Записать decisions/lessons, которых нет в run-файлах - Удалить stale: завершённые планы, устаревшие TODO - Обновить статус проекта в memory-файле 3. SKILLS (если работал с API): - Нашёл gotcha → обновить SKILL.md + дату VERIFIED - Новый API в проекте → создать skill 4. CLEANUP: - Реализованный plan → удалить файл - Stale state.md → убрать "NEXT" / "in progress" - Завершённые TODO в memory → убрать или пометить DONE
Hard-won lesson: stale plan в контексте — AI думает работа не сделана и предлагает её снова. 3+ сессии подряд Claude предлагал batch regen, который давно завершён. Завершённый артефакт опаснее отсутствующего. Cleanup — не опционально.

/compact — команда для ручного сжатия контекста. Используй когда сессия длинная, но задача не закончена. Claude сожмёт старые сообщения, оставив последний контекст. Перед compact — убедись что всё важное в файлах, а не только в чате.

Когда начинать новую сессию:

— Задача завершена → handoff → новая сессия для новой задачи
— Контекст забит (40+ сообщений, модель "тупит") → handoff → новая сессия
— Смена проекта → новая вкладка (не засоряй контекст другим проектом)
— Баг в процессе → новая вкладка для fix (не мешай рабочую сессию)

Когда НЕ начинать новую сессию:

— Середина annotation cycle (контекст плана важен)
— Implement в процессе (Sonnet помнит план и todo)
— Debug с accumulation context (каждый шаг debug добавляет понимание)

↓

Вход в новую сессию (3 шага):
1. Claude читает MEMORY.md → кто ты, что знаешь, какие проекты
2. Claude читает state.md → на какой фазе, что делать дальше
3. Продолжай — как будто не уходил

Если handoff сделан правильно — потери контекста нет. Вообще. Файлы помнят всё. Чат — нет. Поэтому файлы > чат. Всегда.

Deep handoff: для тяжёлых проектов

Стандартный handoff — 4 шага (state, memory, skills, cleanup). Для простых задач хватает. Но в тяжёлых проектах — multi-phase pipeline, сложная архитектура, нетривиальные trade-offs — стандартный handoff теряет нюансы. Новая сессия начинает с чистого контекста и переизобретает решения, которые уже обсуждались.

Deep handoff — расширенный протокол:

Deep handoff (перед /compact или сменой сессии): 1. Стандартный handoff (state + memory + skills + cleanup) 2. Verification cycle #1: Claude перечитывает state.md и все run-файлы → проверяет: ничего не потерял? Все decisions зафиксированы? Все trade-offs записаны? 3. Verification cycle #2: Claude пробует "войти как новая сессия" — читает MEMORY.md → state.md → проверяет: достаточно ли контекста чтобы продолжить без вопросов? 4. Если чего-то не хватает → дописать до /compact

По сути — сухая репетиция handoff. Claude играет роль "новой сессии" и проверяет, хватит ли зафиксированного контекста. Если не хватает — дописывает. Только потом compact. Стоит 30 секунд, экономит часы на переоткрытие контекста.

Когда нужен deep handoff: проект > 3 сессий, сложная архитектура, multi-model pipeline, нетривиальные decisions (которые не очевидны из кода). Для quick fix и однодневных задач — обычный handoff достаточен.

— Collective knowledge: я нашёл gotcha в Soniox, обновил skill, запушил. Коллега получил при следующем sync. Он не наступит на те же грабли.
— Versioned: git history показывает когда что изменилось. Skill сломался после обновления — git diff покажет что поменяли.

Коварность #1: --delete убивает. rsync с флагом --delete удаляет файлы, которых нет в источнике. Коллега создал новый skill локально, но не запушил. Ты запустил sync с --delete — его skill удалён у тебя, а при следующем sync — удалён и у него. Fix: никогда не использовать --delete. Только добавлять, не удалять. Удаление — явное, вручную.
Коварность #2: skills грузятся при старте. Обновил skill через sync — но текущая сессия Claude не видит изменений. Skills загружаются в контекст при старте сессии, не в runtime. Fix: после sync начни новую сессию. Или синкайся в начале работы, не в середине.
Коварность #3: merge conflicts. Два человека обновили один skill одновременно. Git попытается merge через rebase. Если конфликт — rebase абортится. Но Claude Code может зарезолвить сам: зайти в repo директорию, посмотреть конфликтные файлы, объединить оба изменения и завершить rebase. Ручной резолв нужен только если оба изменили одну и ту же строку с противоречащим смыслом — тогда Claude спросит что оставить. На практике — редкость.

Когда синхронизироваться:

— Начало рабочего дня — подтянуть что коллеги запушили вчера
— После правки skill — сразу push, не копить
— После нового API — создал skill → sync → коллега уже имеет
— Не в середине задачи — skills грузятся при старте, sync mid-session бесполезен

Итог: skill sync превращает индивидуальные знания в командные. Каждый gotcha, каждый workaround, каждый новый endpoint — документируется один раз, доступен всем. Это не wiki, которую никто не читает. Это файлы, которые Claude читает автоматически. Написал — все знают.

19. Debugging: когда всё сломалось прямо сейчас

Bug hunting (глава 20) — это про проактивный поиск. Ты сидишь, гоняешь ревью, находишь скрытое. Debugging — другое. Stack trace в лицо. Ничего не работает. Нужно чинить прямо сейчас.

AI отличный дебаггер — если знаешь как его использовать. Проблема: по умолчанию AI пытается угадать причину по описанию. "Наверное дело в X." Нет. Не угадывай — изолируй.

Debugging flow (от быстрого к глубокому):

Шаг 1: Reproduce. "Запусти эту команду. Покажи полный output." Если не воспроизводится — проблема не в коде. Если воспроизводится — идём дальше.

Шаг 2: Isolate. Убери всё лишнее. Минимальный пример, который падает. Чем меньше код — тем виднее баг. Claude хорош в этом: "сократи до минимального воспроизводимого примера".

Шаг 3: Bisect. Если раньше работало — git bisect. Claude может запустить git bisect автоматически: "найди коммит, который сломал тест X". Бинарный поиск по истории — быстрее чтения diff на 50 коммитов.

Шаг 4: Multi-agent debug. Один агент смотрит на stack trace. Другой — на последние изменения. Третий (Gemini) — ищет аналогичный баг в интернете. Параллельно. Кто первый найдёт — тот и прав.
# Реальный пример: Soniox API вернул пустой результат # Шаг 1: Reproduce Claude: "Запусти transcribe с этим файлом" → Пустой JSON. Воспроизведено. # Шаг 2: Isolate Claude: "Попробуй с коротким файлом (10 сек)" → Работает. Значит проблема в размере. Claude: "Попробуй 5 минут" → Пустой. Граница где-то между 10сек и 5мин. # Шаг 3: Bisect (по размеру, не по git) Claude: "ffmpeg — нарежь на 30сек, 1мин, 2мин, 3мин" → 2мин работает, 3мин — нет. Rate limit по длине. # Шаг 4: Gemini background Bash("gemini -p 'Soniox API empty result for files longer than 2 minutes. Known issue?' -o text > /tmp/g-debug.md", run_in_background=true) → Gemini нашёл: нужен async mode для файлов >2мин. Fix за 10 минут вместо 2 часов угадывания.

Ключевой принцип: не давай AI угадывать. AI обучен быть confident — он скажет "проблема в X" с такой уверенностью, что ты поверишь. И потратишь час на fix X, который не был сломан. Вместо этого: reproduce → isolate → данные → fix. Каждый шаг даёт информацию, не гипотезу.

Когда что использовать:
— Stack trace + очевидная ошибка: Opus inline, один промпт. "Вот ошибка, вот код. Fix."
— Работало, перестало: git bisect через Claude. Находит коммит за минуту.
— Непонятная ошибка: multi-agent. Один смотрит код, другой ищет в интернете, третий проверяет skill/API изменения.
— Intermittent bug: сначала reproduce 5 раз подряд. Если не стабильно — race condition или external dependency. Claude может добавить логирование и прогнать 50 раз.

Debugging + skill system: 40% моих багов — не баги в коде, а расхождение между skill и реальностью API. Endpoint поменялся. Rate limit изменился. Новый параметр стал обязательным. Поэтому первое что проверяю при странном поведении API: актуален ли skill? Дата VERIFIED старше недели? Перепроверь. Это не debugging кода — это debugging контекста.

П2.01. Discovery: четыре опоры перед стартом

Boris начинает с research: "Read this folder in depth, understand how it works deeply." У него уже есть codebase, есть контекст, есть задача. Ему нужно только понять код глубже.

Я часто начинаю с нуля. Новая ниша, новый продукт, новый API. Поэтому перед research мне нужен Discovery — зафиксировать 4 вещи, прежде чем тратить время:

1. Что строим — конкретно, одним предложением 2. Для кого — целевой пользователь 3. Что НЕ строим — scope рамка 4. Что считается первым успехом — measurable outcome

Это звучит банально, но без этого AI разъезжается моментально. Ты говоришь "сделай трекер воды", а к концу сессии у тебя социальная сеть с gamification, leaderboard и интеграцией с Apple Watch. AI обожает добавлять scope. Discovery — это scope рамка, которая не даёт этому случиться.

Правило: максимум 1 вопрос за сообщение. Максимум 3 вопроса, если задача простая. Обязательно предложить recommended approach + alternative. Если вход слабый — сказать это прямо.

Discovery делает Opus inline. Не Sonnet — тот соглашается со всем. Не Gemini — тот полезет в интернет до того, как мы определили что ищем. Opus задаёт правильные вопросы и видит слабый вход.

Результат: brief.md (4 опоры) + intent.md (зачем это вообще нужно) + invariants.md (что не должно меняться). Три файла, которые держат рамку на всём протяжении pipeline.

Gemini background
П2.02. Research: Gemini копает, Opus думает

Вот где я принципиально разошёлся с Boris. Он делает research внутри Claude: "Read this folder, study this system." Его research — это чтение кода. Мой research — это чтение интернета, PDF, документации API, скриншотов конкурентов.

Claude не умеет искать в интернете. Он галлюцинирует URL, выдумывает статистику, уверенно цитирует несуществующие исследования. Поэтому для research у меня Gemini CLI — запускается в background через Bash, ходит в интернет, читает PDF нативно, анализирует скриншоты.

Bash( command="source ~/.claude/keys.env && \ GEMINI_API_KEY=$GEMINI_API_KEY gemini \ -p 'Research: [тема]. Find: market size, key players, \ pricing, technical approach, gotchas. \ Be specific with numbers and sources.' \ -m gemini-2.5-pro -o text \ > /tmp/gemini-research.md 2>/dev/null", run_in_background=true )

Gemini приносит сырые данные: ссылки, числа, цитаты, сравнительные таблицы. Но Gemini не думает — он собирает. Думает Opus.

Когда Gemini завершил background task, я читаю его output и передаю Opus для синтеза: "Вот research, вот brief. Какие архитектурные альтернативы? Какие trade-offs? Что выбрать и почему?" Opus видит картину целиком и принимает обоснованное решение.

Паттерн "Gemini → Opus": Gemini = raw material (background). Opus = synthesis и decisions (inline). Это как аналитик приносит данные, а директор принимает решение. Никогда наоборот.

Результат: research.md — не стена текста, а структурированный анализ: что нашли, какие варианты, что рекомендуем и почему. Этот файл — input для Plan.

Это один из самых контринтуитивных принципов. Ты сидишь в Claude Code — зачем тебе другая модель для research? Claude же умный. Claude может всё. Нет. Не может.

Claude не имеет доступа к интернету. Точка. Когда ты спрашиваешь Claude "какой сейчас market size AI code tools?" — он не гуглит. Он вспоминает. Из training data, которая обрезана на определённой дате. И он не скажет "я не знаю" — он уверенно назовёт число. Часто правдоподобное. Иногда правильное. Но ты никогда не знаешь, когда он врёт.

Реальный случай: Claude уверенно процитировал "исследование Gartner 2025" с конкретными числами. Исследование не существовало. Числа были похожи на правду — достаточно, чтобы я построил на них целый research.md. Gemini нашёл реальный отчёт за 30 секунд — числа отличались в 3 раза.

Codex CLI (OpenAI) — та же проблема. GPT-модели не ходят в интернет через CLI. Codex отлично ревьюит код, но для research он бесполезен — он тоже "вспоминает", а не ищет.

Opus — идеален для синтеза research, но не для сбора. Он великолепно анализирует данные, видит trade-offs, строит аргументацию. Но данные ему нужно дать. Без данных — галлюцинации.

Кто реально умеет искать?

Gemini CLI — нативный web search. Когда ты запускаешь Gemini, он ходит в Google, находит свежие источники, цитирует их с URL. Он видит PDF нативно — загрузил, прочитал, структурировал. 2M контекст позволяет скормить ему целую документацию API. И он работает в background — пока Gemini копает, я продолжаю работать с Opus.

Exa.ai — альтернатива для semantic search. Exa ищет не по ключевым словам, а по смыслу: "найди все инди-разработчики, которые написали про свой pipeline разработки iOS apps". Google этого не умеет. Gemini использует Google под капотом — значит, тоже не умеет. Exa закрывает этот gap. На практике Gemini хватает для 90% задач, Exa — для оставшихся 10%.

Паттерн research фазы:

1. Brief готов → формулируем research questions 2. Gemini CLI (background) → raw data: числа, ссылки, цитаты, документация 3. [Если нужно] Exa → semantic search по нишевым источникам 4. Opus (inline) → синтез: "вот 3 альтернативы, вот trade-offs, рекомендую X потому что Y" 5. Результат → research.md (структурированный, с источниками)

Каждый слой делает своё. Gemini/Exa — сборщики. Они приносят сырьё. Opus — аналитик. Он видит картину и принимает решение. Никогда не проси аналитика собирать данные. Никогда не проси сборщика принимать решения.

Почему не один инструмент для всего? Потому что "умный" ≠ "всезнающий". Claude Opus — самая умная модель для рассуждений. Но он слеп без данных. Gemini — посредственный аналитик, но отличный исследователь. Codex — плохой исследователь, но точный ревьюер кода. Специализация > универсальность. Это главный принцип мультимодельной оркестрации.

П2.03. Plan: контракт, не черновик

Boris правильно говорит: "The built-in plan mode sucks. My markdown file gives me full control." Я полностью согласен. plan.md — это файл в проекте, не чат-сообщение и не built-in фича IDE.

Но у Boris plan.md — это единственный документ. У меня plan.md — часть целого run folder:

.kurdyuk-lite/runs/my-project/ ├── state.md ← фаза, текущий шаг, blocker ├── brief.md ← 4 опоры (что / для кого / что НЕ / успех) ├── intent.md ← зачем это вообще нужно ├── invariants.md ← что НЕ должно меняться ├── research.md ← результаты research фазы ├── plan.md ← КАНОНИЧЕСКИЙ IMPLEMENTATION CONTRACT ├── todo.md ← execution surface с verification ├── implementation-readiness.md ← verdict: pass/fail └── final-review.md ← честный verdict финала

plan.md не живёт в вакууме. Он проверяется против brief.md (scope drift?), invariants.md (не нарушаем ли инварианты?), research.md (все ли данные учтены?). Это не один документ — это система документов с перекрёстными проверками.

Boris пишет план и аннотирует его сам. Я тоже аннотирую — но не только я. Plan.md проходит через Annotation Cycle, где его атакуют 4+ агента. Об этом — дальше.

Антипаттерн: тихая подмена scope. AI обожает в plan.md добавлять фичи, которых не было в brief.md. "Давайте ещё добавим кэширование" — когда brief говорит "простой MVP". Поэтому plan проверяется против brief явно.

Multi-agent
П2.04. Annotation Cycle: план атакуют четыре агента

Это самое ценное, что я взял у Boris и расширил. Его идея гениально проста:

Boris Tane: "Claude writes plan.md → I review in editor → I add inline notes → Send Claude back: 'address all the notes, update the document, don't implement yet' → Repeat 1-6 times."

Один человек + один агент. Markdown как shared mutable state. Простая, мощная идея. Но у неё есть ограничение: один взгляд на проблему. Человек видит план через свою призму. Claude соглашается с большей частью (он обучен быть helpful, не critical).

Я добавил мультиагентные проходы. Каждый проход — другой агент, другая перспектива, другие вопросы:

Red Team — атакует план. Ищет scope creep, wrong assumptions, failure modes. "Ты заложил 3 API, но brief говорит только про один. Это conscious decision или drift?" Red Team не предлагает решения — он ломает. Это его работа.

Architect — проверяет структуру. Trade-offs, wrong abstractions, overengineering. "SQLite хватит для этого scope. PostgreSQL — это не техническое решение, это anxiety." Architect думает о том, что будет через 3 месяца, не через 3 часа.

Codex CLI (background) — implementation review. Запускается через Bash в background, читает план и сравнивает с brief. "Phase 3 зависит от API, которого нет в research.md. Вы планируете его добавить или это ошибка?" Codex — прагматик. Его не интересует красота плана, его интересует: это вообще можно реализовать?

Bash( command="codex exec \ 'Review plan.md as implementation expert: \ check feasibility, find pitfalls, \ compare with brief.md for scope drift. \ Be direct.' \ -s read-only \ -C .kurdyuk-lite/runs/my-project/ \ -o /tmp/codex-plan-review.md", run_in_background=true )

Gemini Design (background) — если в задаче есть UI, Gemini смотрит mockups/скриншоты. "Touch target 38pt — минимум 44pt по HIG. Contrast ratio 2.8:1 — нужно 4.5:1." Gemini видит пиксели, Claude видит код. Разные глаза на одну проблему.

Я (User) — продуктовые решения. "Убери регистрацию, сделай проще." Или: "Нет, кэш здесь не нужен, мы это делаем один раз." AI не знает мой бизнес-контекст. Я знаю.

Opus inline
П2.05. Readiness Gate: последний шанс сказать "нет"

Этого нет у Boris. У него после annotation — сразу implement. У меня между ними стоит Readiness Gate — отдельный checkpoint с verdict: pass или fail.

Зачем? Потому что annotation cycle может закончиться на "вроде ок" вместо "точно ок". Readiness Gate — это формальная проверка: все ли dependencies есть? Все ли API researched? Brief не нарушен? Invariants на месте? Todo list готов?

## Implementation Readiness Assessment **Verdict: PASS / FAIL** Checklist: - [ ] brief.md: 4 опоры заполнены - [ ] research.md: все внешние зависимости researched - [ ] plan.md: прошёл annotation cycle (1+ итераций) - [ ] invariants.md: план не нарушает инварианты - [ ] todo.md: задачи разбиты, verification fields пустые - [ ] Нет open questions или blockers **If FAIL:** что нужно доделать перед implement

Это последний шанс сказать "стоп" дёшево. Каждая минута в Implement стоит в 10 раз дороже минуты в Planning. Readiness Gate — это экономия, не бюрократия.

Без pass — код запрещён. Не "попробуем и посмотрим", а именно запрещён. Это правило pipeline.

Sonnet inline
П2.06. Implement: Sonnet кодит, Opus молчит

Boris делает implement через Opus: "Implement it all. Mark completed tasks. Do not stop." Его промпт великолепен в своей прямоте. Но я использую Sonnet, не Opus.

Почему? Потому что к этому моменту вся сложность уже решена. Plan прошёл 4 агента review. Readiness = pass. Todo list готов с конкретными задачами. Реализация должна быть boring — как Boris и говорит: "The creative work happened in the annotation cycles."

Sonnet быстрее, дешевле и не хуже Opus для написания кода по готовому плану. Opus избыточен здесь — его сила в thinking, а thinking на этом этапе уже произошёл.

Правило Boris (которое я использую дословно): "Do not add unnecessary comments or jsdocs. Do not stop until all tasks are completed. Continuously run typecheck." — Минимализм в implement промпте.

Отличие от Boris: я использую todo.md с verification полями. Не просто "mark as completed" — а "mark as completed WITH proof".

## Todo ✅ Реализовать SQLite кэширование **Verify:** 2nd run 0 API calls, response from cache 14ms ✅ Добавить integration tests **Verify:** 23 tests passed, 0 mocks, real SQLite in-memory □ Оптимизировать N+1 запросы **Verify:** ___

Каждая галочка — не "AI сказал done", а "вот proof". Тест прошёл — покажи output. Кэш работает — покажи лог. Запросы оптимизированы — покажи EXPLAIN. Без proof — задача не done.

Multi-layer
П2.07. Review: три слоя проверки

У Boris нет отдельной фазы review после implement. Его implement включает typecheck, и этого ему достаточно — он сам ревьюит код в процессе.

У меня три слоя, и они бегут параллельно:

Слой 1: Codex CLI (background) — code review. Читает написанный код, сравнивает с планом. "Функция deduplicateByTitle не реализована. В plan.md она есть, в коде — нет." Codex видит расхождения между планом и реализацией, которые человек пропустит.

Слой 2: Gemini Visual QA (background) — если есть UI. Сравнивает скриншот реализации с mockup/brief. "Кнопка сместилась на 8px влево. Шрифт заголовка 16pt вместо 18pt." Pixel-level проверка, которую ни один code review не поймает.

Слой 3: Opus Final Review (inline) — честный verdict. Opus читает весь run folder (brief → plan → todo → код) и выносит judgment. "Реализация соответствует плану. Два deferred item из Codex review — некритичные, можно в v1.1."

// Слои 1 и 2 бегут параллельно в background: Bash(command="codex exec 'Review code...' -o /tmp/codex-review.md", run_in_background=true) Bash(command="gemini -p 'Compare screenshot vs mockup...' -o text > /tmp/gemini-qa.md", run_in_background=true) // Когда оба завершились → Opus читает и выносит verdict: Read /tmp/codex-review.md Read /tmp/gemini-qa.md → Opus: final-review.md с verdict

Всё, что нашли Codex и Gemini, фиксируется в final-review.md как "accepted" или "deferred". Ничего не теряется. Если deferred — это explicit decision, не забытый bug.

Вайбкодинг — это не "AI пишет код за тебя"
Это операционная система, где ты — оркестратор. AI = лучший джуниор: делает 80% рутины, но без твоего контекста, вкуса и judgment — результат будет средним. Pipeline не даёт AI врать тебе, phase gates не дают тебе врать себе, а verification не даёт обоим прятать проблемы за галочками.

Код — побочный продукт. Главный продукт — правильные решения.
